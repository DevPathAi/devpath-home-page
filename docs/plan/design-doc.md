## Problem Statement

DevPath AI는 제품(적응형 진단 → AI 맞춤 로드맵 → AI 멘토 → LCS → Q&A 커뮤니티)의 설계·구현이 두텁게 진행됐지만, **하나의 링크로 "이 제품이 무엇이고, 이걸 만드는 사람이 누구이며, 왜 믿을 만한지"를 동시에 전달하는 공개 접점이 없다.**

현재 존재하는 웹 접점은 두 가지 뿐이다.
- `devpath-landing-page` — 48시간 스모크 테스트용 정적 리드 수집 페이지(Google Sheets 파이프라인). 검증 실험 도구지 정식 홈이 아님.
- `devpath-frontend/landing` — Jaspr 빌드 조각.

동남권 본선·예비창업패키지·보육기관 매칭이 임박한 시점에서, 심사위원·투자자·초기 유저 누구에게 링크를 보내도 **제품 가치와 창업자 서사를 한 페이지에서 이해시키는** 정식 홈페이지가 필요하다. 빈 `devpath-home-page` 레포가 그 자리다.

## Demand Evidence

이 세션은 "제품 수요"가 아니라 "홈페이지라는 아티팩트의 필요"를 다뤘다. 홈페이지 수요의 근거:
- 모두의 창업 장기전략 문서(28)가 명시한 본선 차별화 3축 중 두 축이 **공개 표현물**을 전제로 함: "창업자 lived experience 서사 일관성", "이미 만든 결과물(StockPilot/LearnFlow)을 데모 가능한 상태로".
- 기존 `devpath-landing-page`가 이미 리드 수집(이메일·단계·WTP·인터뷰 opt-in)을 돌린다는 것 자체가 "공개 접점에 수요가 있다"는 실동작 증거. 다만 그것은 실험용 최소 페이지라 정식 홈의 역할을 못 함.

한계 인정: "정식 홈페이지가 전환/설득을 실제로 높인다"는 것은 아직 가설이다. 성공 기준(아래)으로 검증한다.

## Status Quo (현재 우회책)

- 링크를 받은 사람은 pitch-deck.html(정적 슬라이드) 또는 스모크 테스트 랜딩 중 하나를 봐야 하고, 둘 다 "제품 기능 + 창업자 신뢰"를 한 흐름으로 주지 못한다.
- 창업자는 매 대상(멘토/심사/유저)마다 맥락을 구두·문서로 재설명한다. 이 반복이 시간 비용.

## Target User & Narrowest Wedge

**듀얼 오디언스(사용자가 명시적으로 "all" 선택)** 이되, 화면 최상단 주 전환은 하나로 고정한다.

- 1차 대상(지금, 본선 타임라인): 심사위원·보육기관·투자자 — "이 사람/이 팀/이 제품을 믿을 수 있는가".
- 2차 대상(상시): 개발 학습자 — "내 문제를 푸는가, 지금 해볼 수 있는가".

**가장 좁은 wedge(주 CTA): "내 실력 진단받기(가입)".** 이유: 진단 시작 하나가 (a) 제품 핵심 가치를 즉시 체감시키고 (b) 실사용 traction 데이터를 만들며 (c) 그 traction이 심사 신뢰의 근거가 되어 두 오디언스를 동시에 만족시킨다. 알파 미완 구간에는 **베타 대기자 등록으로 graceful degrade**(기존 랜딩의 리드 폼 재사용).

## Constraints

- 창업자 1인(부산), 본선/예창패 병행 → **출시 속도가 최우선 제약**.
- 생태계 스택: Flutter Web + Jaspr(Dart), GitOps/k8s, 기존 정적 랜딩(HTML/CSS/JS + Apps Script).
- SEO/공유 미리보기/빠른 로딩 필요(심사위원이 링크로 열고 즉시 판단).
- 개인정보 컴플라이언스 이미 정비됨(PIA/처리방침/동의 마이크로카피 문서 존재) → 리드 폼은 기존 가드(소스코드·토큰·로그 차단) 재사용.
- 코드 없이 배포하면 아무도 못 봄 → 배포 채널이 설계에 포함.

## Premises

1. **별도 존재하되 기존 랜딩 흡수 검토.** 이 home-page는 정식 브랜드·제품 홈. 사용자 선택("기존 랜딩 통합 검토")에 따라, 기존 `devpath-landing-page`의 **리드 수집 기능(폼·Apps Script→Sheets 파이프라인·프라이버시 가드)을 홈에 통합**하고, 스모크 테스트 랜딩은 궁극적으로 홈으로 대체/리다이렉트하는 경로를 설계에 명시.
2. **개인/회사 소개 = 핵심 신뢰 자산.** 장식이 아님. 창업자 lived experience(2년제→학원→독학, StockPilot·LearnFlow)를 서사의 중심 섹션으로. 본선 차별화 축과 정합.
3. **성공 지표는 트래픽이 아니다.** "진단 시작률", "리드 완료 수", "심사/보육기관이 링크 하나로 제품+창업자를 이해했는가(설득까지 스크롤)"가 진짜 KPI.
4. **배포 채널 포함.** 도메인·호스팅·배포 파이프라인을 설계에 포함(아래 Distribution Plan).

## Approaches Considered

### Approach A: 정적 우선 (Minimal Viable) — 베이스
순수 HTML/CSS/최소 JS 단일 사이트. 기존 `devpath-landing-page` 자산(styles, form-utils, Apps Script 파이프라인) 재사용. 진단 CTA는 앱(`devpath-frontend`)으로 링크, 미완 시 대기자 등록.
- Effort: S · Risk: Low
- Pros: 최단 출시 · SEO/로딩 최상 · 리드 파이프라인 즉시 재사용 · 무료 호스팅(CF Pages) · 카피 수정 즉시
- Cons: 앱과 별도 코드/디자인 이원화 · 화려한 인터랙션 한계 · 라이브 지표 수동
- Reuses: 기존 랜딩 HTML/CSS/JS + Apps Script, pitch-deck 섹션

### Approach B: Jaspr 기반 (Ideal Architecture / 생태계 정합)
기존 `devpath-frontend/landing`의 Jaspr(Dart) 위에 정식 홈. 앱과 언어·디자인 토큰 공유, SSR 정적 생성으로 SEO 확보.
- Effort: M/L · Risk: Med
- Pros: 앱과 통합(단일 언어·컴포넌트 재사용) · Jaspr SSR SEO+속도 · 장기 확장성 · Jaspr 빌드 존재
- Cons: Jaspr 러닝커브·성숙도 · 마케팅 사이트를 Dart로 짜는 오버헤드 · 카피 수정마다 빌드 · 1인 시간 부담
- Reuses: Jaspr landing 구조, 앱 디자인 토큰

### Approach C: 하이브리드 — 정적 홈 + 라이브 데모/지표 (Creative)
A의 정적 뼈대 + LCS를 보여주는 인터랙티브 데모 위젯 임베드 + traction 숫자를 Sheets/API에서 자동 fetch해 "라이브 지표"로 노출. 유저="진단 시작", 심사="데모 보기"를 한 페이지에서.
- Effort: M · Risk: Med
- Pros: 속도·SEO 유지하며 제품 즉시 체감 · 라이브 traction 숫자가 심사 신뢰 극대화 · 오디언스별 CTA 공존
- Cons: 데모 위젯 별도 구축·유지 · 라이브 지표 파이프라인 필요 · A보다 복잡
- Reuses: A 전부 + LCS 데모(문서 26 / lcs-data-flow-diagrams.html), pitch-deck

## Recommended Approach

**A→C 단계적 (사용자 확정).**

- **Phase 1 (지금, 출시):** Approach A로 정식 홈을 최단 출시. 목표는 "본선/유저에게 뿌릴 수 있는 진짜 링크" 확보. 기존 랜딩 리드 파이프라인 통합.
- **Phase 2 (fast-follow):** Approach C 요소를 얹음 — (1) LCS 라이브 데모 위젯, (2) traction 라이브 숫자(가입 수·진단 완료 수 등). 심사 설득력을 단계적으로 극대화.
- **B(Jaspr)는 보류:** 앱 통합 이점은 인정하나 현 타임라인엔 오버엔지니어링. 홈 트래픽/유지보수가 커지고 앱-홈 디자인 통합 필요가 실제로 발생하면 그때 재평가.

### 공통 정보구조(IA) — 스크롤 순서
1. **Hero** — 한 줄 가치명제("내 수준에 맞는 다음 단계를 AI가 안내") + 주 CTA "내 실력 진단받기" + 보조 CTA "데모 보기"
2. **문제 서사** — 개발 학습의 "어디서부터 뭘 배워야 할지 모름" 고통(창업자 1인칭과 연결)
3. **핵심 기능 3 + 차별화** — 진단/로드맵/AI멘토 + **LCS 강조 블록**(나란히 라이브 데모, 최상급 표현 없이)
4. **작동 흐름** — 진단→로드맵→AI멘토→Q&A 4스텝 시각화
5. **창업자/회사 소개** — lived experience 서사 + 이전 결과물(StockPilot/LearnFlow) 링크 + "내가 겪었고, 그래서 만든다"
6. **신뢰/traction** — 모두의 창업 진행 배지, 베타 지표(Phase 2에서 라이브화)
7. **요금** — 월 14,900원 + 무료 체험(가설 H1 프레이밍)
8. **리드 캡처** — 베타 대기자/인터뷰 opt-in(기존 폼·가드·Sheets 재사용) + 푸터(연락/제휴/개인정보 링크)

## Success Criteria

- **출시:** Phase 1 정식 홈이 실제 도메인에 배포되어 공유 가능(본선 관련 링크 배포 가능 상태).
- **진단 시작률:** 홈 방문 → "내 실력 진단받기" 클릭률 측정 가능(초기 목표는 baseline 확보).
- **리드 완료:** 기존 스모크 테스트 기준과 연속성 유지(예: 완료 리드 누적, Round 1 기준 ≥20 유지/초과).
- **설득 스크롤:** 심사/보육기관 대상 테스트 뷰어 3~5명이 "제품+창업자"를 스크롤만으로 이해했다고 응답.
- **SEO/공유:** 링크 공유 시 OG 미리보기(제목·설명·이미지) 정상 노출, Lighthouse 성능/접근성 ≥90.

## Distribution Plan

- **호스팅:** Cloudflare Pages(또는 Netlify/GitHub Pages) — 정적 사이트 무료·글로벌 CDN·즉시 배포.
- **도메인:** `devpath.ai`(확인 필요) 연결 + SSL 자동.
- **CI/CD:** `devpath-home-page` 레포 push(main) → 호스팅 자동 배포. 생태계 정합을 원하면 기존 GitHub Actions 패턴 재사용.
- **리드 백엔드:** 기존 `devpath-landing-page`의 Apps Script Web App(Google Sheets)을 그대로 endpoint로 재사용 → 서버 신규 구축 0.
- **기존 랜딩 마이그레이션:** 스모크 테스트 종료 후 `devpath-landing-page`는 홈으로 301 리다이렉트 또는 홈 내 섹션으로 흡수.

## Dependencies

- 기존 `devpath-landing-page` 자산 접근(styles.css, src/form-utils.js, apps-script/Code.gs) — 재사용 원천.
- 앱(`devpath-frontend`) 진단 플로우 라이브 URL(또는 대기자 fallback 결정).
- 브랜드 자산(로고/컬러) — MVP 문서상 외주 예정(로고·컬러 가이드 20만원 항목). 미확정 시 임시 타이포 기반 브랜드로 출시 후 교체.
- 창업자 소개 원고·이력·이전 프로젝트 링크(StockPilot/LearnFlow GitHub 정리 — MVP 액션리스트에 이미 있음).

## The Assignment

**다음 한 가지를 실제로 하라(코드 아님):**

기존 `devpath-landing-page`를 지금 열어, 위 IA 8개 섹션 각각에 **"이 섹션에 들어갈 실제 문장 1~2줄"을 직접 써서 1페이지 아웃라인으로 채워라.** 특히 5번(창업자 소개)은 남의 말이 아니라 당신 목소리로: "2년제→학원→독학"을 겪은 사람이 **왜 이걸 만드는지** 3문장. 이 아웃라인이 있으면 홈페이지 구현은 조립이고, 없으면 아무리 예쁘게 만들어도 빈 껍데기다. 문장을 못 쓰겠는 섹션이 있으면, 그 섹션은 아직 당신도 확신이 없는 것이니 그걸 먼저 표시하라.

## What I noticed about how you think

- 첫 질문에 "all"이라고 답했다. 야망은 좋지만, 홈페이지에서 "모두"는 대개 "아무도"가 된다. 다만 곧바로 "내 실력 진단받기"를 주 CTA로 고르면서 스스로 초점을 잡았다 — 넓게 열되 한 점으로 수렴시키는 판단이 나왔다.
- "기존 랜딩 통합 검토"를 골랐다. 새로 다 짓기보다 이미 돌아가는 리드 파이프라인을 재활용하려는 선택은 1인 창업자의 자원 감각이다. 바닥부터 다시 만드는 함정을 피했다.
- 접근법에서 "A→C 단계적"을 골랐다. 지금 완벽(C)을 좇지 않고 출시 가능한 A로 먼저 링크를 확보한 뒤 설득력을 얹는 순서 — 본선 타임라인이라는 현실 제약을 설계에 반영한 결정이다.
